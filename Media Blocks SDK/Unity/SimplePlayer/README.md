# Unity Simple Player — VisioForge MediaBlocks SDK (net48)

Minimal Unity 6 sample: a `MediaBlocksPipeline` plays a media file
(`UniversalSourceBlock` → `BufferSinkBlock(VideoFormatX.RGBA)`) into a `RawImage`
`Texture2D`, with audio routed to the system default device. Toggle **Use Test Pattern**
in the inspector to swap the file source for a `VirtualVideoSourceBlock` diagnostic
baseline.

## Requirements

- **Unity 6 (6000.x)** — built and verified against 6000.4.6f1.
- **Windows x64** (Editor + Player). Other platforms need their own native runtime.
- **Api Compatibility Level = .NET Framework** — already set in the committed
  `ProjectSettings.asset` (`apiCompatibilityLevel: 3`). The default `.NET Standard 2.1`
  profile is **not compatible** with this SDK build.
- **No system GStreamer in PATH.** If you have a GStreamer install on `PATH`
  (e.g. `C:\gstreamer\1.0\msvc_x86_64\bin`), the player prunes it from the *process*
  PATH at startup so the bundled copy wins — but a clean machine is simplest.

## 1. Build the SDK for Unity

```powershell
cd _SOURCE
./switch-to-unity.ps1
dotnet restore VisioForge.MediaFramework.Solution.Unity.sln
dotnet build VisioForge.MediaFramework.Solution.Unity.sln -c Release
```

Outputs `bin/Release/net48/` per project.

## 2. Copy the project to a short NTFS path

> ⚠️ **Two Windows gotchas:**
> - **MAX_PATH (260 chars):** Unity's `Library/PackageCache/com.unity.test-framework@<hash>/…`
>   paths overflow if the project root is deep. Keep it shallow, e.g. `C:\unity\UnityPlayer\`.
> - **ReFS / Dev Drive:** works, but the project must live on a path Unity can import
>   without `EPERM` rename races. NTFS is the safe default.

Copy `UnityProject/` to e.g. `C:\unity\UnityPlayer\`. The SDK source stays in the
worktree; deploy scripts take an explicit `-UnityProjectRoot`.

## 3. Deploy native + managed binaries

Deploy scripts live in `_SOURCE/` (they auto-resolve the CrossPlatform NuGet and SDK
build artifacts by walking up from `$PSScriptRoot`):

```powershell
cd _SOURCE
./deploy-unity-natives.ps1 -UnityProjectRoot C:\unity\UnityPlayer
./deploy-unity-managed.ps1 -UnityProjectRoot C:\unity\UnityPlayer -Tfm net48
```

**TFM:** Unity 6 with `.NET Framework` compat targets a Mono profile aligned with
.NET Framework 4.8 → `-Tfm net48` (the only supported TFM; netstandard2.1 support is planned).

`deploy-unity-natives.ps1` reads from the **NuGet package cache** (no SDK source needed for
the native runtime). What gets deployed:

| What | From | To |
|---|---|---|
| ~300 GStreamer core libs **+ plugins** + `ca-certificates.crt` (flat) | `…\.nuget\packages\visioforge.crossplatform.core.windows.x64\<ver>\build\x64\` | `Assets/Plugins/x64/` |
| FFmpeg core DLLs + `gstlibav.dll` | `…\visioforge.crossplatform.libav.windows.x64.upx\<ver>\build\x64\` | `Assets/Plugins/x64/` |
| Managed SDK DLLs + deps (Serilog, SkiaSharp, System.Text.Json, …) | SDK build `bin/Release/net48/` | `Assets/Plugins/` |

> **Codecs:** the Core package ships **no FFmpeg decoders** (licensing), so MPEG-4, MP3 and
> other libav-only codecs won't decode without the Libav overlay. The deploy script
> auto-resolves `…libav.windows.x64.upx`; pass `-SkipLibav` to omit it. H.264/H.265 decode
> through the bundled `openh264` / `d3d11` / `nvcodec` plugins without Libav.

> **Why `Plugins/x64/` and not Unity's `Plugins/x86_64/`?** The SDK's native loader expects
> its libs in `<assembly_dir>/x64/`. The player points `SetDllDirectoryW` + process `PATH`
> there. The NuGet package is flat (core + plugins together), so `GST_PLUGIN_PATH` is the
> same folder. **The system PATH is never modified.**

`Assets/Plugins/x64/` is not committed to git — it's regenerated by the deploy script
(see `.gitignore`).

## 4. Run

1. Open `C:\unity\UnityPlayer` in Unity 6.
2. Open `Assets/Scenes/SampleScene.unity`.
3. On the `MediaBlocksPlayer` component (on the RawImage): set **File Path** (default
   `C:\Samples\!video.avi`) and pick an **Aspect Mode** (Stretch / Letterbox / Crop).
4. ▶ Play. The video appears in the `RawImage` (Game view); audio plays through the
   system default device.

## Scripts

Three scripts in `Assets/Scripts/` (the first two are shared verbatim with RTSPViewer):

- **`VisioForgeVideoView.cs`** — reusable RawImage renderer (Unity equivalent of the SDK's
  per-platform VideoView): zero-alloc double-buffered frame upload, vertical flip, and
  Stretch / Letterbox / Crop aspect handling. You never write texture code yourself.
- **`VisioForgeEnvironment.cs`** — one-time GStreamer env setup (PATH/DLL dir/GST_PLUGIN_PATH/
  SSL), `InitializeSdk()` (InitSDK + explicit `Registry.ScanPath`), and the Editor reload
  guard (`StopMainLoop()` on assembly reload / quit).
- **`MediaBlocksPlayer.cs`** — the only sample-specific part (~190 lines): builds
  `UniversalSourceBlock` (`CreateAsync(path, ignoreMediaInfoReader:true)` — the gst-discoverer
  pre-probe fails in this runtime, so decodebin negotiates at build) → `BufferSinkBlock(RGBA)`
  + `AudioRendererBlock`, wires the sink to the video view, and pumps `view.Update()`. A
  `Use Test Pattern` toggle swaps in a `VirtualVideoSourceBlock`.

## Editor lifecycle (important)

The project ships **Enter Play Mode Options → Disable Domain Reload** (required). GStreamer
runs on a native GLib thread Unity can't abort, and `gst_deinit` can't be re-initialized
in-process — so the SDK initializes **once per process** and is **never destroyed on Stop**
(only the per-play pipeline is disposed). A reload guard calls `VisioForgeX.StopMainLoop()`
on script recompile / quit so the domain unloads without hanging. See
[../CLIENT_GUIDE.md](../CLIENT_GUIDE.md) for the full rationale + troubleshooting.

## Known limitations / next steps

- **Windows x64 only** — the bundled CrossPlatform NuGet ships only Windows x64 natives.
- **FFmpeg codecs need the Libav overlay** (deployed by default) — bundled plugins cover
  H.264/H.265 but not MPEG-4/MP3/legacy codecs.
- **Audio** routes to the system default device via `AudioRendererBlock`. Unity
  `AudioSource` integration is TBD.
- **Diagnostic logging** — the scripts still log progress; trim for production.
- **Standalone build** — natives live in `Plugins/x64/`, which Unity does not auto-stage
  into `<game>_Data/` for standalone builds the way it does `x86_64/`. A post-build copy
  step is TBD.
